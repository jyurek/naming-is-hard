Naming is hard

Phil Karlton

There are only two hard things in Computer Science: cache invalidation and naming things.
They're hard because they're not calculable.
Code is communication
The computer doesn't care what you name a class
Even "joke" langauges show that code is written for humans to care about
We program for other people
We work on teams. We need to communicate.
Ideas have power. Names have power.
Giving things names helps you think about them

Sapir-Whorf Hypothesis

A hypothesis holding that the structure of a language affects the perceptions of reality of its speakers and thus influences their thought patterns and worldviews.
We don't work in trivial codebases
Know what you're working with
If you can't name something, it's probably wrong
It might be wrong because you're doing too much
Don't try to contain too much in one label
Don't be too general
"Object" is the only thing that should be "Object"
"Base" is bad
I worked in a project with a base class of "Datum"
WAY too general
And won't someone think of the grammar!
SRP applies to names, too
Something that does one thing will be easier to name

Sandi Metz's rules

  1. Classes can be no longer than one hundred lines of code.
  2. Methods can be no longer than five lines of code.
  3. Pass no more than four parameters into a method. Hash options are parameters.
  4. Controllers can instantiate only one object. Therefore, views can only know about one instance variable and views should only send messages to that object (@object.collaborator.value is not allowed).
Small classes, small methods
